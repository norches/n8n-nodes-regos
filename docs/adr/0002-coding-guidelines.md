# ADR-0002: Coding guidelines and repository conventions

- **Status:** Accepted
- **Date:** 2026-07-18
- **Deciders:** project owner + Claude

## Context and Problem Statement

The package targets n8n **verified community node** status, which imposes hard technical gates, and most code will be written by agentic sessions, which need mechanical, checkable rules rather than taste.

## Decision Drivers

- Verified gates: `npx @n8n/scan-community-package` and n8n ESLint plugins must pass; zero runtime dependencies; provenance publishing.
- Mixed hand-written/generated codebase needs a strict boundary.
- Contributors (human or agent) should never have to guess a convention.

## Considered Options

1. Minimal rules, fix at review time — rejected: verified gates make violations expensive late.
2. **Strict, lint-enforced conventions from day one** — chosen.

## Decision Outcome

### Language and tooling

- TypeScript, `"strict": true` in `tsconfig.json`.
- ESLint with `@n8n/eslint-plugin-community-nodes` (plus legacy `eslint-plugin-n8n-nodes-base` rules where the starter includes them). **No rule may be disabled without an ADR.**
- Prettier with default settings.
- `npx @n8n/scan-community-package` must pass in CI.

### Dependencies

- **`dependencies` in `package.json` stays empty — forever** (verified requirement). HTTP goes through `this.helpers.httpRequest` / `httpRequestWithAuthentication`. Anything else needed at runtime is hand-written or vendored.
- CI asserts `dependencies` is empty.
- Dev-time tooling (codegen, tests, lint) lives in `devDependencies`, including `@n8n/node-cli >= 0.23.0` (required for provenance publishing, mandatory since 2026-05-01).

### Layout and registration

- n8n starter layout: `nodes/<Name>/<Name>.node.ts` + `<Name>.node.json` codex + SVG icon; `credentials/<Name>.credentials.ts`.
- `package.json` `n8n` attribute: `n8nNodesApiVersion: 1`, `strict: false`, `nodes`/`credentials` arrays pointing at compiled `dist/**/*.js`.
  - *Amended 2026-07-18, reverted 2026-07-20:* the package briefly shipped `strict: false` with an `eslint.config.mjs` that appended ignores for the dev-only `scripts/` and `tests/` trees, because the stock config lints `**/*.ts` repo-wide and those files import `vitest`/`node:fs` by nature. That was a mistake: `n8n-node cloud-support` computes eligibility as `isStrictMode && isUsingDefaultConfig` and reports a package in that state as **"NOT eligible for n8n Cloud verification"** — and `n8n.strict` ships inside the published manifest, so the Creator Portal can read it even though `@n8n/scan-community-package` never does. It is the most likely cause of the automated-review failure on 0.1.3.
  - **Current rule:** `strict: true` with a byte-identical stock `eslint.config.mjs`. Dev-time TypeScript lives in `.mts` files (`scripts/generate/index.mts`, `tests/*.test.mts`), which the stock config's `**/*.ts` glob does not match, so no ignores are needed and nothing is suppressed. `npx n8n-node cloud-support` must report **ENABLED**; treat any change that breaks that as a release blocker.
- Package name `n8n-nodes-regos`; `keywords` includes `n8n-community-node-package`.
- Publishing: **GitHub Actions with npm provenance only** — never from a local machine.

### Generated code boundary

- Every generated file starts with `// AUTO-GENERATED by scripts/generate — DO NOT EDIT. Run: npm run generate`.
- Generated files live only in dedicated `generated/` subtrees (`nodes/<Node>/generated/`).
- Codegen never touches hand-written files; hand-written code never edits generated files. Customization happens exclusively through the override layer (`scripts/generate/overrides/**`, [ADR-0005](0005-codegen-pipeline-and-implementation-style.md)).

### n8n UX conventions

- Resource/Operation pattern; operation `name` in Title Case (`Get Many`), `action` in sentence case including the resource (`Get many items`), `description` sentence case.
- Parameter internal `name` values are camelCase and stable across versions.
- Errors are thrown only as `NodeApiError` / `NodeOperationError`, and messages reference the offending parameter's `displayName` where applicable.

### Tests

- Required for all hand-written core: executor, `GenericFunctions` (envelope unwrap, 8213 retry with fake timers, pagination termination, batch serialization incl. capital `Key`), generator invariants (all endpoints mapped, literal path casing preserved incl. `/pos/*` and `/batch`), trigger normalization (both envelope shapes, dedupe, event filtering).
- Not required for generated description data (covered by generator invariant tests + snapshots of 2–3 representative resources).

### Process

- Conventional commits (`feat:`, `fix:`, `docs:`, `chore:`, …).
- Everything in English: code, comments, docs, commit messages.
- Any decision affecting architecture, public surface, conventions, or verified compliance gets an ADR ([docs/adr/README.md](README.md)).

## Consequences

- Good: verified-readiness by construction; zero ambiguity for agents; lint/CI catches violations mechanically.
- Bad: strictness has upfront cost (overrides instead of quick hand-edits to generated files).
